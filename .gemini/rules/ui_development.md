# UI Development & Stitch Design Rules

When interacting with UI designs, specifically those generated by the AI tool "Stitch" or provided via HTML blocks, you must strictly adhere to the following rules:

1. **Persistent TopBar:** The `TopAppBar.jsx` component is locked. Do not override, remove, or drastically alter the `TopAppBar` when implementing new page designs. 
2. **Scoping New Designs:** New HTML designs should only replace the `<main>` content boundary of the view component. Navigation remains global and consistent across the whole application located in `App.jsx`.
3. **Tailwind Framework:** The application natively runs Tailwind CSS v3. All provided HTML classes can be used directly without translation to Vanilla CSS.
4. **Listing and Detail Page Navigation Standard:**
   To ensure consistency, all main entity modules must follow this strictly standardized header button anatomy:
   - **Listing Pages Header (Right-Aligned Order):**
     1. **Delete All:** A square red outline icon button (`<button className="w-8 h-8 rounded border border-error/40 text-error flex items-center justify-center hover:bg-error/10 transition-colors"><span className="material-symbols-outlined text-[18px]">delete</span></button>`). Replace standard label buttons with this icon container.
     2. **Import/Secondary Action:** Standard secondary text button (e.g. `<Button variant="secondary">`).
     3. **Create/New Action:** Standard primary text button (e.g. `<Button variant="primary">`).
   - **Detail Pages Header (Right-Aligned Order):**
     1. **Dynamic Status Action:**
        - When `!isEditing`: A square Back button with outline styling, containing only a `chevron_left` icon and no text (`<Link className="w-8 h-8 rounded border border-primary/40 text-primary flex items-center justify-center hover:bg-primary/10 transition-colors" title="Back"><span className="material-symbols-outlined text-[18px]">chevron_left</span></Link>`).
        - When `isEditing`: A Delete entity button, using the exact square red outline icon format defined for the Listing pages above.
     2. **Edit/Save Toggle:**
        - When `!isEditing`: A primary color Outline button (`className="px-5 py-2 font-bold font-label text-xs uppercase tracking-widest transition-colors border border-primary/50 text-primary hover:bg-primary/10"`). Text = "Edit [Entity]".
        - When `isEditing`: A Solid Filled button (`className="px-5 py-2 font-bold font-label text-xs uppercase tracking-widest transition-colors bg-primary text-on-primary glow-primary"`). Text = "Save [Entity]".
5. **Editable Text Inputs:** 
   When constructing editable text inputs for forms, stats, or values (e.g. max HP, Armor Class), **do not** use standard bordered input boxes. Inputs must rely strictly on a bottom border underline that visually highlights during edit mode. Use or replicate the styling from the `InlineEditable` base (`className="bg-transparent outline-none border-b-[2px] border-primary/40 focus:border-primary"`). Size the input width precisely to the expected data.
6. **Filter Bar Architecture:** 
   When implementing data filters on any listing page, mirror the infrastructure established on the Creature Listing Page. 
   - Wrap the filter block in a defined panel container (`flex flex-wrap gap-4 bg-surface-container-low p-4 border border-outline-variant/10`).
   - Give each distinct filter block structural alignment using column flex and small tracking labels (`text-[10px] text-secondary font-bold uppercase tracking-widest`).
   - Always leverage the custom `<CustomSelect>` component for dropdown constraints, and the `<TraitFilterInput>` for deep tag/array searching logic.
7. **Interactive Roll Affordances & Components:** 
   We maintain strict visual unity for elements that trigger a dice roll (e.g., Saves, Skills, Attacks, Perception).
   - **Standardize With Global Components**: Avoid raw, ad-hoc inline buttons mapping out repetitive mechanics. Use `<StrikeActionGroup />` for a Strike + Multiple Attack Penalty + Damage layout. Use `<StatPill />` for general stats, skills, or attribute tags. 
   - **D20 Icon Affordance**: ANY custom button or element that triggers a dice roll (`onClick={() => rollDice(...)}`) MUST have the `<SingleD20Icon />` component inserted immediately before its typography label. (Note: `<StatPill>` handles its own icon injection natively when `onClick` is provided).
   - **Icon Spinning Animation**: The D20 icon must cleanly rotate on hover using a targeted isolated group. Add `group/btn` (or a similar targeted group name) to the clickable parent container, and strictly apply `<SingleD20Icon className="w-3.5 h-3.5 group-hover/btn:animate-spin opacity-80 shrink-0" />`. Do NOT use a generic `group` and `group-hover` wrapper which will capture hovers uncontrollably from outer layout containers.
   - **Sub-Grouping Restraint**: If an element functionally groups multiple separate but related dice rolls together (like the StrikeActionGroup), only place the D20 icon on the *primary* initiating action of that sequence (e.g., the primary Strike button and the primary Damage button), but omit the icon from subsequent inline modifiers like MAP (-5/-10) penalty buttons to minimize visual clutter.
8. **Action Icons (1-3 Actions, Free, Reaction):**
   When building UI that specifies the action cost of an activity, spell, or item, always use the global `<ActionIcon />` component. Do **not** map out hardcoded SVGs or unicode characters.
   - Example usage: `<ActionIcon action="1" />`, `<ActionIcon action="free" />`, `<ActionIcon action="reaction" />`.
   - The `<ActionIcon />` is built using standard Starfinder 2e geometric scaling, perfectly matching the action diamond format. It uses `currentColor` and is built to sit natively inline via `h-[1em] w-auto inline-block`.
9. **No Dashed/Dotted Borders:**
   NEVER use dashed (`border-dashed`) or dotted borders anywhere in the application styling. The UI theme relies entirely on solid borders or structural background contrasts to define elements (including empty states like unchecked checkboxes or empty slots).
10. **Text Link Interactions:**
    Unfocused raw text links across the application dashboards must default to the secondary red palette baseline (`text-secondary`), transition smoothly up to primary cyan upon user interaction (`hover:text-primary transition-colors`), and inherit an explicit glowing cyan boundary when representing an active state (`text-primary [text-shadow:0_0_8px_rgba(87,230,239,0.5)]`).
